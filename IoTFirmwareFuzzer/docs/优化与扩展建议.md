# IoT Firmware Fuzzer 优化与扩展建议

## 一、Web 界面易用性

| 建议 | 说明 | 优先级 |
|------|------|--------|
| **选项悬停提示** | 鼠标悬停在「协议」「存根模式」等标签时显示简短说明，减少查文档 | 高 |
| **超时倒计时** | 设置超时后，运行中显示「剩余 XX 秒」，结束前 10 秒提示 | 高 |
| **开始前校验** | 未勾存根且未填目标二进制/固件时提示「将无真实目标」；路径填了但文件不存在时警告 | 中 |
| **崩溃列表预览** | 实时统计下方增加「最近崩溃」列表（ID、类型、可利用性），点击可跳转报告或下载输入 | 中 |
| **日志下载** | 提供「下载本次日志」按钮，将当前 log 导出为 .txt | 低 |
| **上次运行摘要** | 停止后保留并显示「上次运行：执行 X 次，崩溃 Y 个，报告 xxx.html」 | 低 |
| **浅色/深色主题切换** | 设置里增加主题切换，照顾不同偏好 | 低 |

---

## 二、功能增强

| 建议 | 说明 | 优先级 |
|------|------|--------|
| **语料库管理** | 在 UI 中查看当前 seeds 目录下种子数量、大小；支持「清空语料库」「从文件上传种子」 | 中 |
| **配置预设/保存** | 保存多组配置（如「MQTT 存根 60 秒」「ARM 固件长时间」），一键加载 | 中 |
| **多 worker 并行** | 配置里支持 workers=4，多进程并行 fuzz 提升吞吐（需改引擎与状态汇总） | 中 |
| **断点续跑** | 停止后可将当前语料库与统计保存；下次选择「从上次继续」加载 | 低 |
| **自定义字典** | 在 UI 中上传或编辑「有趣值」列表（如 0xdeadbeef、超长串），参与变异 | 低 |
| **Power Schedule 选择** | 界面上可选 fast/explore/exploit 等，而不是写死在配置里 | 低 |

---

## 三、稳定性与性能

| 建议 | 说明 | 优先级 |
|------|------|--------|
| **语料库定期修剪** | 语料超过 N（如 10000）时自动按优先级修剪，避免内存与磁盘膨胀 | 高 |
| **启动前环境检查** | 未勾存根时检查 QEMU/Binwalk 是否在 PATH；缺失时明确提示并给出安装建议 | 高 |
| **Web 单例** | 同一时刻只允许一个 fuzz 任务（已实现），可再加「任务名/ID」便于多会话区分 | 低 |
| **日志行数上限** | 内存中只保留最近 500 条（已实现），可做成可配置或按大小限制 | 低 |

---

## 四、报告与分析

| 建议 | 说明 | 优先级 |
|------|------|--------|
| **崩溃详情页** | 报告或单独页面：每个崩溃展示 backtrace、触发输入十六进制、exploitability 说明 | 高 |
| **覆盖率趋势图** | 报告中增加「执行次数–覆盖率」或「时间–崩溃数」简单折线图（前端图表库） | 中 |
| **报告列表** | UI 中「历史报告」列出 output/reports 下所有 HTML，可选打开而非仅最新 | 中 |
| **导出 CSV** | 将崩溃列表导出为 CSV（ID、类型、PC、输入路径、时间）便于用 Excel 分析 | 低 |
| **与 GDB 集成** | 当目标为本地二进制时，崩溃后可选「用 GDB 分析」自动拉取 backtrace 写入报告 | 低 |

---

## 五、部署与运维

| 建议 | 说明 | 优先级 |
|------|------|--------|
| **Docker 镜像** | 提供 Dockerfile：内置 Python 依赖 + 可选 QEMU，一键 `docker run` 起 Web UI | 中 |
| **Web 认证** | 当绑定 0.0.0.0 时可选简单 HTTP Basic 或 token，防止未授权访问 | 低 |
| **环境变量配置** | 通过环境变量指定配置文件路径、输出根目录、Web 端口等，便于容器/脚本化 | 低 |

---

## 六、核心引擎（进阶）

| 建议 | 说明 | 优先级 |
|------|------|--------|
| **AFL++ 兼容模式** | 直接调用 afl-fuzz 或复用其 QEMU 插桩，共享语料与队列，便于与现有 AFL 工作流结合 | 低 |
| **更多协议模板** | 增加 Modbus、DNP3、BACnet 等工控/楼控协议的结构化变异模板 | 低 |
| **响应引导权重** | 黑盒时根据设备响应（如 CONNACK 码、延迟）调整种子能量，而不仅靠「新信号」 | 低 |

---

以上按「高/中/低」优先级排序，可先做高优先级的 UI 提示、倒计时、崩溃预览、语料修剪和环境检查，再视时间加报告与部署相关项。
